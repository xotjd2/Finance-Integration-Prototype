# 금융 IT 통합관리시스템 기획서

## 1. 프로젝트 개요

- 프로젝트명: 금융 IT 통합관리시스템
- 프로젝트 유형: 웹 기반 운영 대시보드 프로토타입
- 배포 구조: 프론트엔드 `Vercel`, 백엔드 API `Railway`

본 프로젝트는 보험 및 금융권에서 운영되는 다양한 대내외 인터페이스를 한 화면에서 조회하고 관리할 수 있도록 설계한 운영 지원 시스템이다. 운영자는 인터페이스 상태, 처리 현황, 장애 징후, 재처리 이력, 프로토콜 분포를 통합적으로 확인할 수 있으며, 신규 인터페이스 등록과 설정 수정, 장애 재처리까지 단일 화면에서 수행할 수 있다.

기존 운영 환경에서는 인터페이스별 관리 지점이 분산되어 있어 장애 인지와 대응이 늦어지고, 상태 이력과 변경 내역을 직관적으로 파악하기 어렵다. 본 시스템은 이러한 문제를 개선하기 위해 "운영 현황 집계 + 인터페이스 목록 + 상세 관리 + 이벤트 가시화"를 하나의 운영 콘솔로 통합하는 것을 목표로 한다.

## 2. 추진 배경 및 필요성

- 금융/보험권 인터페이스는 REST API, SOAP, MQ, 배치, SFTP 등 이기종 프로토콜이 혼재한다.
- 운영 담당자는 장애 여부뿐 아니라 재시도 정책, 대상 기관, 최근 실행 시각, 처리량, SLA 수준까지 함께 판단해야 한다.
- 기존 방식은 모니터링 화면과 운영 메모, 설정 정보가 분리되어 있어 즉시 대응이 어렵다.
- 웹 기반 통합 화면은 운영 정보 접근성을 높이고, 관리 흐름을 단순화하는 데 유효하다.

## 3. 목표

### 3.1 주요 목표

- 운영 인터페이스 현황을 단일 대시보드에서 통합 조회
- 장애, 경고, 처리 중, 정상 상태를 직관적으로 시각화
- 인터페이스 등록, 수정, 삭제, 재처리 기능 제공
- 프로토콜 분포, 성능 추이, 운영 타임라인 제공
- 웹에서 실제 동작 가능한 프로토타입 구현

### 3.2 기대 효과

- 운영자가 장애 징후를 빠르게 식별하고 대응 가능
- 인터페이스 관리 정보의 중앙화
- 업무 흐름을 하나의 화면으로 통합
- 향후 DB, 인증, 실시간 모니터링 연동으로 확장 가능한 기반 확보

## 4. 대상 사용자

- 대외계 운영 담당자
- 금융 IT 운영 관제 담당자
- 인터페이스 개발 및 유지보수 담당자

## 5. 핵심 기능

### 5.1 대시보드 요약

- 전체 운영 인터페이스 수
- 오늘 처리 건수
- 평균 SLA
- 장애/경고/실행중 건수
- 최근 장애 브리핑

### 5.2 운영 인터페이스 목록

- 인터페이스명, 기관, 프로토콜, 상태, 최근 실행 시각 표시
- 상태 배지로 정상/경고/실패/실행중/중지 구분
- 항목 선택 시 상세 정보와 편집 폼 연동

### 5.3 상세 관리 기능

- 인터페이스 신규 등록
- 기존 인터페이스 수정
- 인터페이스 삭제
- 장애 인터페이스 재처리

### 5.4 운영 시각화

- 프로토콜 분포
- 시간대별 평균 응답시간 추이
- 운영 타임라인
- 최근 장애 및 알림 이슈

### 5.5 데모용 상태 시뮬레이션

- 운영 인터페이스 상태는 10초마다 일부 항목이 임의로 변동되도록 구성
- `정상 -> 경고 -> 실패` 흐름을 화면에서 시각적으로 확인 가능
- 상태 변화와 장애 감지 흐름을 짧은 주기로 관찰 가능

## 6. 차별성

- 단순 목록형 CRUD가 아니라 운영 대시보드 관점으로 구성
- 금융권 인터페이스 특성을 반영한 프로토콜과 운영 메타데이터 설계
- 정적 화면이 아니라 실제 API와 연동되는 동작형 프로토타입
- 프론트/백엔드를 분리 배포해 서비스 운영 구조를 명확히 제시

## 7. 화면 구성

### 7.1 메인 대시보드

- 상단 KPI 카드
- 장애 브리핑 패널
- 프로토콜 분포 차트
- 성능 추이 그래프
- 운영 이력 타임라인

### 7.2 인터페이스 운영 패널

- 운영 인터페이스 목록
- 상태 안내 배너
- 항목별 재처리 버튼
- 선택된 인터페이스 상세 및 편집 폼

## 8. 기술 구성

- 프론트엔드: `Next.js 16`, `React 19`, `TypeScript`, `Tailwind CSS 4`
- 백엔드: `Spring Boot 3.5`, `Java 17`
- 데이터 저장 방식: 프로토타입 단계에서는 메모리 기반 데이터 관리
- 배포:
  - 프론트엔드: `Vercel`
  - 백엔드 API: `Railway`

## 9. 배포 구조

본 프로젝트는 프론트엔드와 백엔드를 각각 클라우드에 배포하는 구조를 사용한다.

- 사용자 접속: Vercel에 배포된 웹 UI
- 데이터 조회/수정: Railway에 배포된 Spring Boot API
- 프론트엔드 환경변수 `NEXT_PUBLIC_API_BASE_URL`로 Railway API 주소 연결
- 백엔드는 CORS 설정으로 `localhost:3000` 및 `https://*.vercel.app` 접근 허용

이 구조를 통해 웹 UI와 관리 API를 독립적으로 운영할 수 있으며, 프론트와 백엔드의 역할도 명확하게 분리된다.

## 10. 향후 확장 방향

- DB 연동을 통한 영속 데이터 관리
- 사용자 로그인 및 권한 체계 추가
- 실시간 이벤트 스트리밍(WebSocket/SSE) 도입
- 알림 채널(SMS, 메일, 메신저) 연동
- 배치 스케줄러 및 인터페이스별 이력 저장 강화
- 운영 통계 리포트 다운로드 기능 추가

## 11. 결론

금융 IT 통합관리시스템은 복잡한 인터페이스 운영 현황을 단일 웹 화면에서 통합 관리할 수 있도록 설계된 프로토타입이다. 단순 조회를 넘어 등록, 수정, 삭제, 재처리, 장애 가시화까지 포함하여 운영 시나리오를 실제에 가깝게 구성했으며, Vercel과 Railway를 활용한 분리 배포 구조를 통해 웹 UI와 백엔드 API를 함께 운영할 수 있도록 설계하였다.
