# 금융 IT 통합관리시스템 개발문서

## 1. 문서 목적

본 문서는 금융 IT 통합관리시스템 프로토타입의 기술 구조, 개발 범위, 실행 방법, API 구성, 배포 방식, 향후 개선 포인트를 정리한 개발 기준 문서이다.

## 2. 시스템 개요

프로젝트는 프론트엔드와 백엔드가 분리된 구조의 웹 애플리케이션이다.

- 프론트엔드: 운영 대시보드 UI, 인터페이스 목록 및 편집 화면 제공
- 백엔드: 대시보드 데이터 조회, 인터페이스 CRUD, 재처리 API 제공
- 데이터 계층: 프로토타입 단계에서는 메모리 기반 컬렉션 사용

## 5. 아키텍처

### 5.1 전체 구조

```text
사용자 브라우저
   ↓
Vercel 배포 Next.js 프론트엔드
   ↓ fetch
Railway 배포 Spring Boot API
   ↓
메모리 기반 인터페이스 데이터
```

### 5.2 동작 흐름

1. 사용자가 웹 UI에 접속한다.
2. 프론트엔드는 `NEXT_PUBLIC_API_BASE_URL` 환경변수를 사용해 백엔드 API를 호출한다.
3. 백엔드는 Railway에서 실행되며 `/api` 하위 엔드포인트를 제공한다.
4. 응답 데이터는 대시보드, 목록, 상세 패널에 렌더링된다.
5. 등록/수정/삭제/재처리 요청은 REST API로 처리된다.

## 7. 도메인 모델

주요 모델은 다음과 같다.

- `ManagedInterface`: 인터페이스 기본 정보와 운영 상태를 보관
- `InterfaceConfig`: 엔드포인트, 인증 방식, 페이로드 형식, 재시도 정책 등 세부 설정 보관
- `Incident`: 최근 장애/이슈 정보
- `TimelineEvent`: 운영 이력 정보
- `ProtocolSummary`: 프로토콜별 분포 요약
- `PerformancePoint`: 시간대별 성능 추이 값

## 8. API 명세

기본 경로: `/api`

### 8.1 대시보드 조회

- Method: `GET`
- Path: `/dashboard`
- Description: 대시보드 요약, 인터페이스 목록, 장애 목록, 프로토콜 분포, 성능 추이, 운영 타임라인을 함께 반환

### 8.2 인터페이스 목록 조회

- Method: `GET`
- Path: `/interfaces`
- Description: 운영 인터페이스 목록 반환

### 8.3 인터페이스 생성

- Method: `POST`
- Path: `/interfaces`
- Description: 신규 인터페이스 생성

### 8.4 인터페이스 수정

- Method: `PUT`
- Path: `/interfaces/{id}`
- Description: 특정 인터페이스 설정 수정

### 8.5 인터페이스 삭제

- Method: `DELETE`
- Path: `/interfaces/{id}`
- Description: 특정 인터페이스 삭제

### 8.6 인터페이스 재처리

- Method: `POST`
- Path: `/interfaces/{id}/retry`
- Description: 해당 인터페이스를 재처리하고 상태를 정상으로 복구

## 10. 로컬 실행 방법

### 10.1 백엔드 실행

```bash
cd backend/demo
./gradlew bootRun
```

```powershell
cd backend\demo
.\gradlew.bat bootRun
```

- `http://localhost:8080`

### 10.2 프론트엔드 실행

`frontend/.env.local` 파일에 아래 값을 설정한다.


```bash
cd frontend
npm install
npm run dev
```

기본 실행 주소:

- `http://localhost:3000`

## 11. 배포 구조

### 11.1 프론트엔드 배포: Vercel

- `frontend` 디렉터리를 기준으로 배포
- 빌드 명령: `npm run build`
- 실행 환경: Next.js
- 환경변수:
  - `NEXT_PUBLIC_API_BASE_URL=https://<railway-domain>/api`

Vercel은 웹 UI 배포를 담당하며, 백엔드 API와 분리된 프론트엔드 운영 환경을 제공한다.

### 11.2 백엔드 배포: Railway

- `backend/demo` 디렉터리를 기준으로 배포
- Gradle 기반 Spring Boot 애플리케이션 실행
- `PORT` 환경변수를 Railway가 주입하며, 애플리케이션은 `server.port=${PORT:8080}` 설정으로 이를 사용한다.
- 필요 시 `APP_CORS_ALLOWED_ORIGIN_PATTERNS` 환경변수에 실제 Vercel 도메인을 추가 설정할 수 있다.

Railway는 Spring Boot API 실행 환경을 담당하며, 외부에서 접근 가능한 API 엔드포인트를 제공한다.

## 12. 현재 제약사항

- DB가 아닌 메모리 저장 방식이라 서버 재시작 시 데이터가 초기화된다.
- 사용자 인증/권한 기능이 없다.
- 상태 변화 일부는 데모용 시뮬레이션이다.
- 장애 이력 및 운영 로그가 영구 저장되지 않는다.
